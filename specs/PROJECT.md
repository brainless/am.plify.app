# am.plify.app: Technical Overview

We are creating a web application with a backend app and a browser app.

## Backend Application

I prefer a backend application built with Rust, Actix Web, ts-rs (to generate TypeScript types for the RESTful API), SQLite for database, LiteStream for automatic backups to S3 compatible storage.

Use environment variables and .env files for local, test or live variables for browser app root URL, DB path, any keys for analytics, Sentry, etc.

## Browser Application

I prefer a browser application build TypeScript, SolidJS, Solid Router, Tailwind CSS, Vite, npm which works with the TypeScript types generated for the API from backend. All marketing related content will use Server Side Rendering.

Use environment variables and .env files for local, test or live variables for backend app root URL, any secrets, keys for analytics, Sentry, etc.

## Hosting

The backend will be hosted on Scaleway. I prefer using Terraform and Ansible to manage a 4 GB Linux server. We use the domain `api.plify.app` to host the backend. We should have nginx and `www.plify.app` should redirect to `am.plify.app`. We should use Let's Encrypt for SSL and have end to end HTTPS enabled in CloudFlare - I will manually set it in their control panel. The browser app will be hosted on CloudFlare Pages on `am.plify.app` and I will set it up manually.

## Emails and notifications

We need in app notifications (with history) in the browser app. We need email notification. We will use Resend for all transactional emails.

## Questions for Clarification

### Authentication & User Management

- **User Registration**: Should we implement email verification for new user signups?

We should use email verification when use does not use social login. Our backend needs to support both types and the needed flows (forgot password, reset password, social login and add social login when you already have accot with email/password). Email is mandatory even for social login since we need to send transactional emails.

- **Password Requirements**: Any specific password complexity requirements?

8 characters minimum, mix of alphabets, numbers and special character mandatory. All checks done in backend.

- **Social Login**: Do you want to support OAuth login (Google, Twitter, GitHub)?

We should support social login using Google, Apple, Twitter, LinkedIn and GitHub

- **User Roles**: Are there different user types beyond founders/business owners (e.g., admins, moderators)?

Only one role for now - founder, which represents founders/business owners.

### Points & Rewards System

- **Point Values**: How many points should different types of help activities be worth?

Each task/activity will be well defined and have a point associate. We can create a relevant file in `specs/` folder for this, to record the activity label, description, points. For our MVP, we will list only tasks that can be done online and are global in nature (social media engagement for example). We read this in our backend. TypeScript scripts should be generated by backend by reading this file. Backend (API) and browser (UI) app should use this.

**Backend file**: YAML format at `specs/help_activities.yaml` (text-heavy with descriptions)

- **Point Limits**: What should the daily limit be for earning points?

A user can complete 3 tasks per day at a maximum. Each task's point is determined from the file mentioned before.

- **Point Usage**: Can points be redeemed for anything, or are they just for reputation?

Every user starts with 99 points. When somone helps them, the point(s) are deducted from them and awarded to the helping user. So points are always exchanged.

- **Dispute Process**: How should point disputes be handled? Manual review by admins?

We will build a community driven review process. I will describe my thoughts separately. This is not needed in MVP.

### Content & Moderation
- **Screenshot Verification**: Should screenshots be automatically processed or manually reviewed?

Screenshots are kept only for record, show per submission but there is no processing or verification needed. We trust each user who has helped and submitted. When we build the review process, we will enabled taking away points or banning users.

- **Content Moderation**: Do we need automated content filtering for inappropriate submissions?

Not for the MVP.

- **Reporting System**: Should users be able to report fake or inappropriate help submissions?

We will add such features in the review mechanism separately.

- **Geographic Verification**: How do we verify users are actually in the vicinity for local services?

For the MVP, the list of possible tasks will not have anything related to geography, thereby avoiding a verification process.

### Product/Service Listings
- **Approval Process**: Should new product/service listings require approval before going live?

No. Our comminity based review process will tackle this in future.

- **Categories/Tags**: Do you have a predefined list of business domain tags?

**Backend file**: YAML format at `specs/business_domains.yaml` (text-heavy with tag labels)

- **Media Uploads**: Should product listings support images, videos, or just text and links?

Just text and links for now. Links should each be labeled from the system, like "Instagram account". We need a file in `specs/` for all such allowed form fields - this will drive our API and UI.

**Backend file**: YAML format at `specs/product_form_fields.yaml` (text-heavy with field labels and descriptions)

- **Trial Access Management**: How should the free trial access request system work technically?

Any founder of a Product/Service has to specify they will share a special trial access. This shows up as a buttom to request access to trial in that Product/Service's page. Any helping user can request access, a notification is sent to the owner of the Product/Service. The owner of the Product/Service can send DM to respond to each request. DM notification is sent to the helping user.

Basically we need a Direct Messaging feature and surface it only through Product/Service Trial Access for now. DMs are associated with the Product/Service along with the users involved.
